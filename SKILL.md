# Project Operating

使用此 Skill，让软件项目从想法到上线保持可执行、可验证、可恢复，并以仓库文件为依据。把文件系统视为项目状态来源，但重要结论必须结合代码、测试、构建结果和可运行界面验证。

## 核心原则

- 新项目从模糊想法到上线，按主线推进：`orient → requirements → visualize → engineer → deploy → handoff`。
- 已有项目要继续，先 `orient`，再根据真实状态跳到正确阶段，不机械从头开始。
- 修 bug 直接走 `debug`；改方向走 `change`；只问进度走 `status/orient`，默认不改文件。
- 不要用聊天历史替代项目事实。重要状态、决策、任务和验证结果必须沉淀到项目根目录文档。
- 文档服务于下一步行动。不要为小任务强行创建没有实际帮助的仪式化文档。

## 项目操作文档

在项目根目录维护这些文件，按需创建：

- `ProjectState.md` - 当前阶段、当前目标、已确认决策、当前任务、验证状态、下一步。第一个读，第一个更新。
- `Requirements.md` - 已明确的产品需求。新项目默认用它承载需求梳理结果。
- `Design.md` - 已确认视觉方向、界面原型规则、交互、动效和设计系统规范。有界面项目进入开发前必须有。
- `Tasks.md` - 可执行任务列表、用户验收方式、技术验证和实施日志。
- `Decisions.md` - 只追加不覆盖的关键决策记录。
- `Deployment.md` - 部署平台、环境、验证、监控、回滚、成本、安全和隐私信息。
- `Summary.md` - 里程碑、项目收尾或交接总结。

可选补充文件：

- `Concept.md` - 只有在想法非常早期、需要单独记录概念探索时创建。
- `Plan.md` - 只有在需要更完整产品/技术规划时创建。
- `Spec.md` / `API.md` / `DataModel.md` - 只有在接口、数据模型、业务流程或外部契约需要明确约束实现时创建。

## 运行模式

根据用户请求推断模式。不确定时使用 `normal`。

- `strict` - 重大规划产出后、不可逆操作前都需要确认。适用于高风险、需求模糊或对用户影响大的项目。
- `normal` - 只在关键决策点确认。常规实现、验证和状态更新可以连续推进。
- `fast` - 对已确认计划直接执行，不额外规划。用户说“实现”“继续”“直接做”，或明确跳过确认时使用。

如果用户只是询问状态或分析问题，不要修改文件，除非用户要求更新，或必须通过更新项目状态才能避免后续接力出错。

## 第一步：orient

任何项目任务开始前，先定位状态：

1. 检查项目根目录是否存在 `ProjectState.md`、`Requirements.md`、`Design.md`、`Tasks.md`、`Decisions.md`、`Deployment.md`、`Summary.md` 以及可选补充文件。
2. 如果是已有项目，读取足够的代码和配置来验证文档。不要只相信过期任务勾选。
3. 将用户请求归类为：`requirements`、`visualize`、`engineer`、`deploy`、`debug`、`change`、`status` 或 `handoff`。
4. 简短告诉用户当前进入哪个动作，然后继续。

`orient` 输出要说明：真实阶段、已完成内容、缺失内容、阻塞、下一步。如果用户要求继续且下一步明确，直接进入对应阶段。

## 主线阶段

### requirements：需求梳理与明确

用于把用户的模糊想法转化为确定、可描述、可设计、可开发、可验收的产品需求。

使用时机：

- 用户只有产品想法，需要从 0 开始。
- 现有项目需求不清，导致设计或开发无法判断。
- 用户改变产品方向，需要重新明确范围。

执行方式：

- 当用户需求模糊时，不要直接脑补并生成最终 `Requirements.md`。必须先通过引导式对话把需求逐步澄清。
- 每轮只问一个主题，最多 3 个问题。优先问会影响产品方向、MVP 范围、核心流程或验收标准的问题。
- 能给选项时给选项，降低用户回答成本；用户选择后先总结已确认信息，再进入下一轮。
- 无法确认但为了推进需要采用的内容，必须标为“假设”；会影响后续设计或开发的问题，必须标为“待确认”。
- 只有当需求达到可设计、可开发、可验收的程度，或用户明确要求基于假设继续，才生成或更新正式 `Requirements.md`。
- 更新 `ProjectState.md`，将下一步指向 `visualize` 或 `engineer`。
- 在 `strict` 模式或高风险项目中，等待用户确认需求后再进入视觉化或开发。

引导式澄清流程：

1. Raw Intent Capture：复述用户原始想法，说明当前理解。
2. Gap Detection：指出最影响下一步的需求缺口。
3. Guided Clarification：分轮提问，每轮围绕一个主题。
4. Requirement Synthesis：把已确认内容、假设和待确认项合成为需求草案。
5. User Confirmation：请求用户确认、修正或授权基于假设继续。

需求澄清主题库：

- 动机：为什么要做这个项目？
- 问题：要解决什么具体问题？
- 目标用户：谁会使用？在什么场景下使用？
- 方案：准备用什么产品方式解决？
- 方案理由：为什么这种方式比其他方式更合适？
- 现有方案/竞品：用户现在怎么解决？现有方案哪里不够好？
- 深层洞察：这个需求背后真正重要的判断是什么？
- MVP 范围：第一版必须做什么？哪些可以延后？
- 非目标：当前明确不做什么？
- 验收标准：用户如何判断它做对了？
- 最大不确定性：哪些问题不解决会影响设计或开发？

示例对话方式：

```text
我先按当前信息理解为：你想做一个面向交易者的 AI 复盘工具，帮助用户从交易记录或截图中发现问题。

现在最影响下一步的是目标用户和输入方式。我先问一个问题：
这个产品第一版主要服务哪类用户？
A. 日内交易者
B. 波段/中长期交易者
C. 刚开始学习交易的新手
```

`Requirements.md` 建议结构：

```markdown
# Requirements

## Raw Intent
用户最初如何描述这个想法。

## Product Goal
产品要达成什么结果。

## Target Users
目标用户是谁，使用场景是什么。

## Core Problems
要解决的具体问题。

## User Flows
核心用户流程。

## Functional Scope
MVP 必做功能和后续功能。

## Non-Goals
当前明确不做什么。

## Acceptance Criteria
用户如何判断需求被正确实现。

## Assumptions
为了推进而暂时采用的假设。

## Open Questions
仍未确定但可能影响后续阶段的问题。

## Confirmation
用户确认记录，或用户授权基于哪些假设继续。
```

完成标准：需求具体到可以进入视觉化或开发；关键不确定项已解决、记录假设，或明确标为阻塞；用户已确认需求，或明确授权基于记录的假设继续。

### visualize：视觉化、交互与设计规范

用于把文字需求转成可看的界面概念，让用户判断产品需求是否正确，并沉淀后续开发必须遵守的设计规范。

有界面项目进入开发工程前必须经过 `visualize`。纯 API、CLI、SDK、后端服务可以跳过视觉界面，改为接口/使用体验设计，并在 `ProjectState.md` 记录跳过原因。

执行方式：

1. Visual Intake：获取视觉意图。
   - 如果用户有明确想法，提取参考风格、色彩、字体、信息密度、动效、品牌约束。
   - 如果用户没有想法，根据产品类型、目标用户和使用场景推导 2-3 个视觉方向供用户选择。
2. Visual Direction：确定视觉方向。
   - 记录产品气质、风格关键词、信息密度、色彩方向、字体方向、动效强度和可访问性要求。
3. Interface Prototype：实现或生成可浏览的核心界面原型。
   - 覆盖核心页面、关键状态、空状态、加载、错误、成功、权限或边界场景。
4. Interaction & Motion：确认交互和动效。
   - 明确 hover、active、disabled、loading、empty、error、success 等状态。
   - 动效只服务于反馈、层级和流程理解，不做无意义装饰。
5. Design System：从已确认原型反向沉淀 `Design.md`。
   - 不要凭空写一套与原型不一致的设计规范。
6. User Validation：让用户基于可见界面判断需求和设计是否正确。

当用户没有视觉想法时，使用这样的选择方式：

```text
A. 专业工具型：信息密度高，干扰少，适合高频使用。
B. 现代 SaaS 型：清晰、友好、上手快，适合通用业务产品。
C. 内容/记录型：强调阅读、输入和长期沉淀，适合日志、知识库、复盘类产品。
```

`Design.md` 建议结构：

```markdown
# Design

## Visual Direction
视觉方向和选择理由。

## Pages / Views
页面、视图和状态清单。

## User Flows
核心用户流程在界面中的体现。

## Layout Rules
页面结构、网格、间距、信息密度。

## Color System
主色、中性色、语义色和使用规则。

## Typography
字体、字号层级、行高、字重。

## Components
按钮、输入、上传、卡片、弹窗、Toast 等组件规范。

## Interaction States
hover、active、disabled、loading、empty、error、success。

## Motion Principles
动效强度、持续时间、使用场景和禁用场景。

## Accessibility
对比度、键盘可用性、减少动效偏好。

## Design Constraints
后续新增界面必须遵守的规则。
```

完成标准：用户能看到主要页面或核心流程的视觉原型；`Design.md` 已从原型反向沉淀；用户确认可以进入开发工程。

### engineer：开发工程实现

用于真正编写代码，并让技术用户和非技术用户都能判断开发是否有效。

执行方式：

1. 根据 `Requirements.md` 和 `Design.md` 拆分可验收任务。
2. 必要时创建 `Spec.md`、`API.md` 或 `DataModel.md`，约束接口、数据和业务规则。
3. 实现一个任务后，运行最相关的技术验证。
4. 在 `Tasks.md` 记录用户验收视角和技术验证视角。
5. 更新 `ProjectState.md` 的阶段、验证状态和下一步。
6. 如果实现偏离需求或设计，先走 `change`，更新 `Decisions.md`、`Requirements.md` 或 `Design.md`，再改代码。

`Tasks.md` 任务记录必须包含：

```markdown
- [x] 任务名称
  - Status: Done
  - 用户可验证结果：用户现在可以做什么。
  - 用户验收方式：用户如何判断它是否正确。
  - 技术验证：运行了什么测试、构建、lint、浏览器检查、curl 或 CLI 命令。
  - 已知限制：还没完成、只是 mock、或需要后续处理的内容。
  - Log:
    - YYYY-MM-DD：实施记录和结果。
```

验证策略：

- Web 应用：运行构建或 dev server，验证主路由和本次改动的核心流程。
- API：运行端点检查或测试，验证状态码、请求/响应结构和关键错误路径。
- CLI/工具：执行命令并检查 stdout/stderr 和退出码。
- 后端服务：运行服务和关键集成路径，检查日志和失败场景。
- 文档/规划任务：验证目标、决策、任务和下一步没有矛盾。

完成标准：任务已实现，用户知道如何验收，技术验证有结果，已知限制已记录。

### deploy：上线部署

用于把产品从可运行实现推进到真实可访问、可维护的线上状态。

部署前必须提示用户确认：

- 部署平台：Vercel、Netlify、Railway、Fly.io、自有服务器等。
- 是否需要自定义域名。
- 生产环境变量有哪些，由谁提供。
- 数据库、对象存储和迁移放在哪里。
- 第三方 API key、支付、邮件、AI 服务等外部依赖。
- 是否涉及用户隐私、敏感数据或权限控制。
- 是否需要日志、监控、备份、回滚和成本限制。

执行方式：

- 创建或更新 `Deployment.md`。
- 配置构建命令、运行时、环境变量、数据库/存储、域名和 SSL。
- 部署后做线上验证，不要只看平台显示成功。
- 更新 `ProjectState.md`、`Tasks.md` 和必要的 `Decisions.md`。

`Deployment.md` 建议结构：

```markdown
# Deployment

## Platform
部署平台和项目标识。

## Production URL
线上地址。

## Build & Runtime
构建命令、运行时版本、启动命令。

## Environment Variables
变量名称和用途，不写真实密钥值。

## Data & Storage
数据库、对象存储、迁移方式、备份策略。

## Domain & SSL
域名、SSL、DNS 状态。

## Validation
上线后验证过的核心路径。

## Monitoring
日志、错误追踪、性能监控。

## Rollback
回滚方式。

## Costs & Limits
平台成本、额度、速率限制。

## Security & Privacy
认证、权限、敏感数据和第三方 API 风险。
```

完成标准：线上地址可访问，核心路径验证通过，环境和回滚信息已记录，用户知道部署后的风险和维护事项。

### handoff：总结与交接

用于项目暂停、里程碑收尾、交给另一个 Agent，或用户要求总结。

执行方式：

- 更新 `ProjectState.md`，写清真实阶段、当前目标、验证状态、阻塞和下一步。
- 必要时创建或更新 `Summary.md`。
- 包含已完成内容、未完成内容、关键决策、已知风险、验证结果、部署信息和第一个下一步动作。

## 横向动作

### debug

当用户报告行为异常、测试失败、构建错误或部署问题时使用。

- 可行时先复现或检查失败，再改代码。
- 修复最小且完整的根因。
- 用暴露问题的同一路径验证修复。
- 更新 `Tasks.md` 的用户验收方式和技术验证。
- 只有当修复改变架构、范围或契约时，才记录 `Decisions.md`。

### change

当用户改变方向、范围、优先级、技术方案、工作流、视觉方向或已确认决策时使用。

- 重大实现前先追加 `Decisions.md`。
- 按顺序更新：决策记录 → `ProjectState.md` / `Tasks.md` → `Requirements.md` / `Design.md` / 规格文档 → 代码。
- 保留历史上下文，不要把过去决策直接抹掉。

`Decisions.md` 使用：

```markdown
## YYYY-MM-DD - 简短决策标题

### 背景
是什么触发了这个决策。

### 决策
之后项目应以什么为准。

### 替代方案
如果有，记录考虑过的相关选项。

### 影响
受影响的文档、任务、代码、风险或后续工作。
```

### status

当用户只问进度、当前阶段、还剩什么、是否能上线时使用。

- 读取项目文档和代码事实。
- 必要时运行构建、测试或轻量验证。
- 汇报真实状态，不默认改文件。
- 如果文档明显过期，先说明不一致，再询问是否需要更新状态文档；低风险且用户要求继续时可以直接更新。

## ProjectState.md 模板

```markdown
# Project State

## Current Stage
Requirements / Visualize / Engineer / Deploy / Maintaining

## Active Objective
当前迭代要达成什么。

## Confirmed Decisions
- Decision:

## Current Tasks
- [ ] Task

## Validation Status
- Last checked:
- Result:

## Blockers
- Blocker:

## Next Step
新 Agent 接手后应该先做的第一个动作。
```

## 护栏

- 有界面项目进入开发前，必须完成视觉化确认和 `Design.md`，除非用户明确要求跳过并记录原因。
- 不要把 `Requirements.md` 写成空泛愿景；它必须能支持设计、开发和验收。
- 不要把 `Design.md` 写成抽象规范；它必须来自用户确认过的界面原型或明确视觉方向。
- 不要把任务标为完成，除非用户可验收结果和技术验证都已记录，或用户明确接受纯文档完成。
- 不要在重大需求、视觉、架构或部署决策不清时继续硬做。问一个聚焦问题，或在低风险情况下明确记录假设。
- 不要记录真实密钥值。部署文档只记录环境变量名称、用途和配置位置。
